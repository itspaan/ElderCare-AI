# Model Card — ElderCare AI Disease Screening Model

A short, structured description of the machine-learning model used for health
screening in ElderCare AI. Format follows the common "model card" convention
used in ML research and industry.

---

## 1. Model details

| Item | Value |
|---|---|
| **Model name** | ElderCare disease screening model |
| **Type** | Random Forest Classifier (supervised, multi-class) |
| **Library** | scikit-learn (`RandomForestClassifier`) |
| **Hyperparameters** | `n_estimators=100`, `random_state=42` (all others default) |
| **Artifact** | `models/disease_model.pkl` — a pickled dict `{"model": ..., "features": [...]}` |
| **Training script** | `training/train_model.py` |
| **Inference location** | `predict_disease_from_vitals()` in `core/agent.py` |

The pickled `features` list defines the column order used at inference. Training
and inference must use the same order — keep them in sync.

---

## 2. Intended use

- **Purpose:** screen for a *likely* chronic condition common in Taiwan's
  elderly population, from a few basic vitals and symptoms, as a conversational
  aid for elderly users and their caregivers.
- **Intended users:** elderly people and caregivers, via the ElderCare AI chat.
- **Out of scope / must NOT be used for:** medical diagnosis, treatment
  decisions, or any clinical action. Every result is paired with a disclaimer
  to consult a healthcare professional. **This is a screening aid, not a
  diagnosis.**

---

## 3. Inputs (features)

| Feature | Type | Description | Normal reference |
|---|---|---|---|
| `Age` | int | Patient age in years | 60–95 (elderly cohort) |
| `Systolic_BP` | int | Systolic blood pressure (mmHg) | ~120; high > 140 |
| `Blood_Sugar` | int | Fasting blood sugar (mg/dL) | 70–110; high > 126 |
| `Joint_Pain` | 0/1 | Reports joint pain | 0 = absent |
| `Memory_Loss` | 0/1 | Reports memory loss / confusion | 0 = absent |
| `Fatigue` | 0/1 | Reports persistent fatigue | 0 = absent |

## 4. Outputs (classes)

`Hypertension`, `Diabetes`, `Osteoarthritis`, `Dementia`, `Healthy`.

The tool returns the predicted class, a confidence (max class probability),
the **key factors** that drove the prediction (see §7), and standard care advice.

---

## 5. Training data

- **Source:** synthetic dataset, generated by `data/generate_dummy.py`
  (`data/elderly_synthetic_data.csv`).
- **Size:** 2,000 rows. Split 80 / 20 (train 1,600 / test 400), stratified by class.
- **Class distribution:**

  | Class | Count |
  |---|---|
  | Hypertension | 628 |
  | Diabetes | 510 |
  | Osteoarthritis | 405 |
  | Healthy | 277 |
  | Dementia | 180 |

- **How it is generated:** each row is assigned a condition by weighted random
  choice, then its vitals/symptoms are sampled from ranges chosen to match that
  condition (e.g. Hypertension → `Systolic_BP` 140–180; Diabetes → `Blood_Sugar`
  126–250 and `Fatigue=1`; Dementia → `Memory_Loss=1` and older age).

---

## 6. Evaluation

On the held-out test set (400 samples):

| Metric | Value |
|---|---|
| Accuracy | **1.00** |
| Macro F1 | **1.00** |
| Per-class precision / recall / F1 | 1.00 across all five classes |

### ⚠️ Honest interpretation (read this — it matters for the thesis)

**The 100% accuracy is a property of the synthetic data, not evidence of a strong
model.** The generator maps each condition to clean, non-overlapping feature
ranges with no noise or comorbidity, so the classes are perfectly separable. The
Random Forest is effectively re-learning the generation rules.

This is **expected and acceptable for a prototype**, but it means accuracy on
synthetic data does *not* predict real-world performance. This limitation is the
primary motivation for collecting **real survey data** (see
[DATA_COLLECTION.md](DATA_COLLECTION.md)) and re-evaluating. Report this honestly;
a perfect score on self-generated data is a known anti-pattern and a committee
will expect you to acknowledge it.

## 7. Explainability

For each prediction the model produces the top factors that drove the result
(`_explain_prediction()` in `core/agent.py`). Method:

1. Score the predicted class's probability for the actual input.
2. For each feature, reset it to a clinical **baseline** (BP→120, blood sugar→90,
   symptoms→absent, age→65) and re-score.
3. The drop in the predicted class's probability is that feature's
   **contribution** (positive = pushed toward the prediction).
4. Report the top factors by absolute contribution, with direction.

This is a **perturbation-based, per-prediction** explanation — it needs no
retraining and no extra dependencies, and is straightforward to defend. A more
formal alternative (future work) is SHAP `TreeExplainer`.

Example output: *"blood pressure (158) raised this result; age (72) raised this result."*

---

## 8. Limitations & ethical considerations

- Synthetic data → no real-world validation yet (see §6).
- Small feature set; real conditions involve comorbidities and many more signals.
- Not a medical device; no regulatory clearance. Screening aid only.
- Possible harm if a user treats output as diagnosis → mitigated by the mandatory
  disclaimer in every health response.

## 9. Maintenance

- Retrain: `python -m training.train_model` (reads the CSV, writes the `.pkl`).
- When adding a feature/class, update `data/generate_dummy.py`, the `FEATURES`
  list in `training/train_model.py`, the tool signature in `core/agent.py`, and
  `_FEATURE_BASELINE` / `_FEATURE_LABELS` for explainability.
